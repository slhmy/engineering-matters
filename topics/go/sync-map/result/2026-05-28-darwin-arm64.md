# Result: 2026-05-28 darwin/arm64

This result is one local observation. It is useful for showing trends, but it does not mean every machine, Go version, or application will produce the same ordering.

If the allocation column shows two `B/op` values, the format is `-cpu=1/-cpu=8`.

## Environment

- Date: 2026-05-28
- System: darwin/arm64
- Go: go1.26.1
- Command:

```bash
cd topics/go/sync-map/benchmark
go test -bench=. -benchmem -benchtime=1s -cpu=1,8
```

## Stable Keys, Read-Heavy

1,000 keys, 99% reads, 1% writes:

| Implementation | `-cpu=1` | `-cpu=8` | Allocation |
| --- | ---: | ---: | --- |
| `mutex` | 7.845 ns/op | 79.82 ns/op | 0 B/op, 0 allocs/op |
| `rwmutex` | 7.712 ns/op | 43.81 ns/op | 0 B/op, 0 allocs/op |
| `syncmap` | 13.66 ns/op | 3.051 ns/op | 0 B/op, 0 allocs/op |
| `shard32` | 8.572 ns/op | 18.53 ns/op | 0 B/op, 0 allocs/op |

Observations:

- On one CPU, regular lock-based solutions are cheap, and the extra machinery of `sync.Map` is more visible.
- With eight CPUs, the read path of `sync.Map` shows a clear advantage.
- The sharded map also reduces single-lock contention, but it still pays for hashing and shard-level locking.

Observations:

- As the write ratio rises, `sync.Map` starts allocating.
- Under multi-core writes, both single-lock strategies, `mutex` and `rwmutex`, queue noticeably.
- When keys are evenly distributed, the sharded map spreads write contention.

| `shard32` | 8.115 ns/op | 24.69 ns/op | 0 B/op, 0 allocs/op |

Observations:

- Hot keys did not hurt the read-path advantage of `sync.Map` in this benchmark.
- The effectiveness of a sharded map depends on whether hot keys actually land on different shards; the more concentrated the hot keys are, the easier it is for sharding to lose value.

## Continuous New-Key Inserts

Every operation inserts a new key:

| Implementation | `-cpu=1` | `-cpu=8` | Allocation |
| --- | ---: | ---: | --- |
| `mutex` | 126.8 ns/op | 240.3 ns/op | 55/46 B/op, 0 allocs/op |
| `rwmutex` | 129.7 ns/op | 252.4 ns/op | 58/45 B/op, 0 allocs/op |
| `syncmap` | 316.3 ns/op | 94.88 ns/op | 120/122 B/op, 3 allocs/op |
| `shard32` | 128.8 ns/op | 80.99 ns/op | 57/73 B/op, 0 allocs/op |

Observations:

- New-key inserts trigger underlying map growth, so every option has additional byte cost.
- `sync.Map` allocates more when inserting new keys; this workload is no longer only about its stable read path.
- With eight CPUs, both `sync.Map` and the sharded map reduce single-lock queueing, but the sharded map allocates less.

## Local Conclusions

This run supports a few small conclusions:

- `sync.Map` is strong when the workload is read-heavy, keys are stable, and many goroutines read concurrently.
- On one CPU or under low concurrency, simple `map + lock` solutions may already be fast enough.
- As the write ratio rises, look at both `ns/op` and `allocs/op`.
- A sharded map is a controllable but more complex option, useful when keys are well distributed and write contention is visible.
- These numbers describe the shape of this benchmark only. They should not be copied directly into application decisions.
